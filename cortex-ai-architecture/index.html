Overview

Cortex embeds a customer-facing AI agent with significant domain expertise in Walmart supplier analytics. The agent is built on Claude (Anthropic API) and derives its intelligence from five layered knowledge sources — not from model fine-tuning. Each layer can be updated independently without retraining or redeploying the model.

The architecture is designed so that:

  • Platform-level knowledge (Walmart domain, data interpretation) is maintained by the DuitLabs team and versioned like code
  • Brand-specific knowledge is maintained by the customer and updated in real time via the Brand Settings UI
  • Guardrails and quality controls wrap the full stack and operate on every request/response cycle
Key principle: The model is a commodity. The expertise lives in the prompt layers. A carefully authored system prompt with well-structured domain knowledge outperforms a fine-tuned model in this use case — and can be updated in hours, not weeks.

Architecture diagram

The five layers are nested — each wraps the one inside it. The innermost layer is the foundation model. The outermost is the guardrails ring that monitors every request and response.

The five layers of the Cortex AI stack. Inner layers are more stable; outer layers change more frequently. Brand context (Layer 4) is customer-managed.

The five layers

1
Foundation model
Claude API · Anthropic

The base model provides general reasoning, language understanding, and code capabilities. DuitLabs does not fine-tune the model. All domain expertise is injected through the layers above.

What lives here: system prompt (static persona and behavioral rules), conversation history (rolling window), tool definitions (functions the agent can call to retrieve data), and RAG retrieval for relevant documentation.

Update cadence: Anthropic manages model improvements. DuitLabs controls the system prompt and tool definitions, versioned in source control. Key decision: use the API, not a self-hosted model — fine-tuning is reserved as a future lever if the prompt-based approach hits a hard ceiling.

2
Walmart domain knowledge
DuitLabs-managed · versioned

The agent's fluency in Walmart data and retail operations. A general-purpose model does not know how to interpret Walmart data (e.g. OTIF, WOS, In-Stock Rate) or what those metrics mean in practice. This layer encodes that expertise explicitly — including how to call and interpret data from the Scintilla API. Note that this layer covers Walmart platform knowledge; brand-specific context (what makes a particular supplier's situation unique) is handled in Layer 4.

Scintilla API (Basic & Charter) Retail math Data interpretation rules Buyer context Terminology glossary

Implementation: The walmart-merchant-analyst skill — a structured markdown document injected into the system prompt at session start. Maintained by DuitLabs and versioned alongside application code. The skill includes a full glossary of Walmart terminology so the agent uses terms correctly.

Note from initial testing: The model required explicit guidance on how to interpret Walmart data. This is expected — the solution is to encode interpretation rules in plain language in this layer, not to fine-tune the model.

3
Platform skills
Cortex-managed · modular

Behavioral capabilities that sit above domain knowledge and shape how the agent reasons, structures its responses, and communicates. These are patterns, not facts.

Merchant analyst Data analyst Communication Response format rules

Each skill is a modular markdown document loaded only when relevant — the communication skill loads for message drafting, not for pure data analysis. Updated when customers report quality issues or new use cases emerge. No engineering sprint required.

Platform playbooks are also defined at this layer — default behaviors for common situations that apply to all brands unless overridden. For example, a platform executive summary playbook might produce a bulleted response covering the current week compared to the prior year. A brand can override this with their own version at Layer 4. The brand playbook always takes precedence.

Sub-agent orchestration

Some tasks require specialized work that is cleanly separated from the customer-facing response. Cortex handles this through sub-agents — purpose-built agents that the primary agent can invoke, wait on, and incorporate without the customer ever knowing a handoff occurred.

The pattern: the walmart-merchant-analyst is always the primary agent and the only voice to the customer. When it needs data it cannot reason about on its own — raw Scintilla queries, aggregations, period-over-period calculations — it invokes the walmart-data-analyst as a sub-agent via a tool call. The data analyst runs its analysis and returns structured results. The merchant analyst then synthesizes those results into a response and replies to the customer. The customer sees one coherent answer, with no visibility into the internal handoff.

1
Customer message
Received by the merchant analyst
2
Merchant analyst decides it needs data
Invokes the data analyst as a tool call, passing the specific data question
3
Data analyst runs
Queries Scintilla, runs calculations, returns structured results to the merchant analyst
4
Merchant analyst synthesizes and responds
Incorporates data analyst output into a single coherent reply — one voice to the customer throughout

Why this separation matters: data retrieval and analysis is a distinct skill from commercial interpretation and communication. Keeping them as separate agents means each can be updated, tested, and improved independently. The merchant analyst's skill document never needs to describe how to run a query — it only needs to know how to interpret and communicate the result. This also makes the system easier to debug: if the answer is wrong, you can quickly identify whether the failure was in data retrieval (data analyst) or in interpretation and framing (merchant analyst).

Additional sub-agents (future): the same pattern can be extended to other specialized tasks — a document drafting agent for generating formatted buyer presentations, or a forecasting agent for scenario modeling. Each plugs in as a tool call to the merchant analyst and returns its output invisibly to the customer.

4
Brand context
Customer-managed · real-time

The brand-specific layer. This is what makes the agent feel like it knows the customer's brand — not generic Walmart expertise, but expertise applied to Widget Inc. specifically. Managed through the Brand Settings UI and injected on top of everything else at session start.

4a — Context notes

Persistent facts about the brand carried as background knowledge in every session.

Evergreen Time-bound Past
  • Evergreen — stable facts with no expiry: buyer preferences, inventory posture, competitive context
  • Time-bound — active for a defined period, then archived: current promo, seasonal program, event window
  • Past — notes from prior periods, injected alongside active notes but labeled as historical context. The agent uses them for period-over-period analysis — understanding a tough comp, a one-time event, or a prior promo — without treating them as current facts. Default window: last 24 months.

Notes older than a configurable threshold (default: 6 months) are flagged in the UI. The agent is instructed to treat stale notes with lower confidence and surface when a note may be outdated.

4b — Brand playbooks

Brand-specific playbooks that override the platform defaults defined in Layer 3. When a brand has a playbook for a given situation, it always takes precedence over the platform version. If no brand playbook exists for a trigger, the platform playbook runs as the default.

Trigger detection uses the model's own language understanding — no separate classifier required. The agent checks brand playbooks first, then falls back to platform playbooks if no brand override matches.

Example: The platform executive summary playbook produces a bulleted response covering the current week compared to the prior year. A brand that prefers prose and a monthly cadence creates their own executive summary playbook with those instructions. From that point on, the brand's version runs instead of the platform default — same trigger, different output.
ScenarioWhat runs
Brand has a playbook matching the trigger Brand playbook runs. Platform playbook is ignored.
Brand has no playbook for this trigger Platform playbook runs as the default.
No playbook at either level matches Agent responds using general domain knowledge and skills — no playbook active.

Injection format — brand override active:

### Active playbook: Executive summary (brand override)
[Brand: Widget Inc. overrides platform default]
The user has requested an executive summary. Apply Widget Inc.'s format:
- Write in prose, not bullets
- Frame as a monthly update, not a weekly snapshot
- Lead with the top commercial story, followed by in-stock and velocity context
- Close with one recommended action for the coming month

Injection format — platform default active:

### Active playbook: Executive summary (platform default)
The user has requested an executive summary. Apply the standard format:
- Bulleted summary, ≤5 points
- Focus on current week vs. same week prior year
- Always include a recommended action
5
Guardrails & evals
Wraps every request

Input guardrails

  • Scope enforcement — reject or redirect questions outside the Walmart supplier context
  • Single-brand isolation — never surface data or context from another brand in the session
  • Prompt injection defense — detect and block attempts to override system instructions
  • PII handling — strip or flag sensitive personal data before it reaches the model

Output guardrails

  • Data grounding check — numerical claims must be supported by the injected session data; the agent must never fabricate metric values
  • Confidence signaling — stale context notes or ambiguous data trigger uncertainty surfacing, not false confidence
  • Citation linking — responses surface which brand context notes influenced them. Playbooks and platform skills are not cited in the UI since they are not visible to brand or broker users.

Evals suite

A battery of test prompts with expected outputs, run before any change to layers 2 or 3 ships. Each eval is a JSON object with prompt, injected_context, expected_behavior, and failure_modes.

Eval categoryWhat it checks
Data interpretationCorrectly reads velocity, in-stock rate, and OTIF from a given snapshot
Retail mathCalculates maintained margin, WOS, and GMROII correctly
Playbook triggeringBuyer meeting playbook fires correctly; stays dormant when trigger is absent
Context note injectionApplies brand's buyer preferences when drafting a buyer email
Scope enforcementCorrectly declines competitor brands or non-Walmart data
Staleness handlingFlags stale context note rather than asserting its contents as current fact
Tone and formatResponse matches platform communication standard: concise, action-oriented, Walmart-fluent

Session assembly — how the context is built

At the start of every Cortex session, the following is assembled and injected into the model context in order. Playbook detection is the exception — it runs per turn, not at session start.

1

2
Session context injection dynamic, per session
User role and Scintilla access tier (Basic / Charter) Active brand name, broker, categories, buyers, distribution Live data snapshot — current week's metrics from PostgreSQL Brand context notes — evergreen + active time-bound (Layer 4a)
3
Conversation history rolling window
Last N turns of the current session (typically 20–30 turns)
4
User message + playbook check per turn
Playbook trigger detection runs against the user's message Matching playbook(s) prepended to user message context (Layer 4b)
Token budget: At Charter tier with a full context note library, injected context is estimated at 3,000–6,000 tokens before conversation history. Claude's 200K context window provides ample headroom. Context notes are short by nature — capacity and cost are not a concern at the scale of typical brand usage.

Observability

Observability is what makes the rest of the system improvable. Without it there is no reliable way to know what users are actually asking, whether the agent is performing well, or which parts of the knowledge stack are doing useful work. Three distinct areas need to be tracked.

Usage and behavior

The foundation layer of observability — understanding how the product is being used in practice. Every session and turn should be logged with enough context to answer basic operational questions without having to dig through raw model output.

SignalWhat it tells you
Questions asked per brand per sessionWhich brands are most active; whether usage is concentrated in a few heavy users or distributed broadly
Query topic distributionWhat users are actually asking — buyer meeting prep, in-stock analysis, promo performance, etc. Drives roadmap priorities and skill investment
Playbook trigger frequencyWhich playbooks are firing most often; which are never triggered and may need their trigger condition revised or removed
Context notes cited per sessionWhich notes are actively influencing responses vs. sitting idle. Notes that are never cited across many sessions are candidates for review or removal
Sub-agent call rateHow often the merchant analyst is invoking the data analyst; whether certain query types consistently require a data sub-call
Token usage per session per brandCost visibility and the primary input for setting and calibrating token cap thresholds
Session length and turn countWhether users are getting answers quickly or requiring multiple follow-ups — a proxy for response quality

Response quality

Usage data tells you what is being asked. Quality signals tell you whether the answers are actually good. These are harder to collect automatically — quality requires judgment — so the approach combines explicit user feedback, implicit behavioral signals, and periodic human review.

Explicit feedback

A thumbs up / thumbs down on each agent response, surfaced inline in the chat. Low friction for the user — one click. This is the primary quality signal and the most actionable: a thumbs down with an optional free-text comment is enough to triage whether the failure was in data retrieval, interpretation, or communication format.

A simple rating is more valuable than it sounds. Over time, the pattern of thumbs-down responses reveals systematic gaps in the knowledge stack — if buyer meeting prep responses consistently get poor ratings, that points directly to the buyer meeting playbook needing work.

Implicit signals

Behavioral patterns that suggest a response was insufficient without the user explicitly saying so:

  • Immediate rephrasing — the user asks the same question a different way in the next turn, suggesting the first answer didn't land
  • Short session after a complex query — user asked a substantive question and then left; may indicate dissatisfaction or confusion
  • Follow-up clarification requests — "what do you mean by..." or "can you explain..." after a response

Human review

A weekly sample of sessions — ideally 10–20 across different brands and query types — reviewed by a DuitLabs team member against a simple rubric: was the data correct, was the interpretation sound, was the format appropriate? This is the highest-signal quality check and the one most likely to catch subtle failures that user ratings miss. Review sessions should be logged and any failures triaged immediately through the update process.

The feedback loop

Observability is only valuable if it feeds back into the system. The goal is a closed loop where signals from real usage continuously improve the knowledge stack:

1
Signal captured
Thumbs down rating, implicit behavioral signal, or human review flags a poor response
2
Triage to layer
Wrong data → data analyst sub-agent or Scintilla integrationWrong interpretation → Layer 2 domain knowledge or Layer 3 skillWrong format or tone → platform or brand playbookMissing brand context → Layer 4 context note gap
3
Fix and add eval
Edit the relevant skill document; add an eval test case that would have caught the failure
4
Deploy and monitor
Run eval suite, merge, redeploy; watch for recurrence of the same failure pattern
The feedback loop is the compounding advantage. Every failure caught and fixed makes the system permanently smarter. A team that runs this loop well will outperform a team with a better initial model but no improvement process.

Keeping the agent up to date

Observability surfaces the failures. This section describes how to fix them. The goal is a process where an issue caught through ratings, human review, or a customer report can be triaged, fixed, and deployed the same day — without engineering involvement for most cases.

When the agent gets something wrong

1
Triage to the layer — wrong data interpretation → Layer 2; wrong tone or format → Layer 3; didn't apply brand context → Layer 4 injection bug; fabricated a number → Layer 5 guardrail gap
2
Edit the skill document — add the missing rule or clarify the ambiguous one. For Layer 4 failures, check the injection format and session assembly order.
3
Add an eval test case — write the test that would have caught this failure before it reached a customer.
4
Run the eval suite — confirm the fix works and doesn't break existing behavior.
5
Merge and redeploy — skill document changes deploy as part of normal application deployment.
Most customer-reported issues are Layer 2 or Layer 3 failures — domain knowledge gaps or communication style mismatches. These can be fixed with a skill document edit. No engineering sprint required.

Model upgrades

When Anthropic releases a new model version, run the full eval suite against the new version before switching. Model upgrades occasionally shift behavior in subtle ways that require skill document adjustments — the eval suite is the safety net.

Storage, versioning, and deployment

Skills and playbooks are different types of content that are stored, updated, and deployed differently. Understanding the distinction matters for how engineering builds the data model and deployment pipeline.

Platform skills

A platform skill is a single markdown document that contains everything for that skill — persona, rules, interpretation guidance, glossary, and embedded platform playbooks. There is no concept of adding an individual piece in isolation. When anything changes, the whole document is updated and redeployed as one unit.

1
Edit the skill file
Add the new rule, update the playbook, clarify the interpretation guidance — directly in the markdown document
2
Review and version
Change is committed to source control — full history, diff view, rollback available at any time
3
Run eval suite
Automated tests confirm the change works as intended and hasn't broken existing behavior
4
Deploy
Skill file ships as part of normal application deployment — no model retraining, no separate pipeline
Adding a new platform playbook means editing the skill document and adding the playbook block in the right section. Adding a new data interpretation rule is the same — edit, commit, eval, deploy. The entire skill is always deployed as a unit, so there is no risk of a partial update where some rules are live and others aren't.

Brand playbooks

Brand playbooks are stored in the database as records attached to a brand, not in source-controlled files. Each playbook is a discrete record with a title, trigger, instructions, and type. Adding or editing a brand playbook does not touch the platform skill at all — it is closer to editing a context note than deploying a code change.

Important for engineering: because brand playbooks are live immediately on save, there is no eval gate between an edit and production. This is fine in v1 when DuitLabs authors all brand playbooks. When self-service is opened up to brands, consider a lightweight review step or a preview mode so brands can test a playbook before it goes live.

Context notes

Context notes follow the same model as brand playbooks — stored in the database per brand, live immediately on save, no deployment step. The difference is that context notes are facts, not instructions. Any user with brand access can add or edit them. The database should maintain an audit log of who added or changed a note and when, both for accountability and to support the "confirm still accurate" staleness prompt in the UI.

Future considerations

Fine-tuning (not recommended for v1). If the prompt-based approach hits a hard ceiling on domain expertise, fine-tuning on a curated dataset of high-quality Cortex responses is the next lever. This should not be pursued until the prompt layers are fully built out and evaluated — the ceiling is likely higher than expected.

Retrieval-augmented generation (RAG). For brands with large context note libraries or attached documents, RAG enables the agent to retrieve only the most relevant context per turn rather than injecting everything at session start. Relevant when the context note library exceeds ~20–30 entries per brand.
